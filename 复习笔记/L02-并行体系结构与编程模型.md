## 二、核心知识点

### 2.1 编程模型(Software)vs 执行模型(Hardware)(P14,★考点)

| 概念 | 含义 | 例子 |
|------|------|------|
| **编程模型**(Programming Model,软件) | 程序员如何描述应用,即程序员角度看到的机器模型 | 顺序模型(von Neumann)、数据并行(SIMD)、数据流模型、多线程模型(MIMD、SPMD) |
| **执行模型**(Execution Model,硬件) | 硬件底层如何执行代码 | 乱序执行、向量机、数据流处理机、多处理机、多线程处理机 |

**关键结论**:执行模型与编程模型可以差别很大。
- 顺序模型可以在**乱序执行**的处理器上执行;
- **SPMD** 编程模型可以用 **SIMD** 处理器实现(GPU 即如此)。

### 2.2 Flynn 分类(Flynn's Taxonomy,1972)(P15–P20,★重点)

依据**指令流(Instruction Stream)**如何作用于**数据流(Data Stream)**进行分类。流(stream)= 处理器执行或操作的一系列项(指令或数据)的集合。

**缩写说明**:CU = Control Unit(控制单元)、PU = Processing Unit(处理单元)、MU = Memory Unit(存储单元)、IS = Instruction Stream(指令流)、DS = Data Stream(数据流)、PE = Processing Element(处理元)、LM = Local Memory(本地存储)。

| 类型 | 全称 | 含义 | 特点 | 典型体系结构/机器 |
|------|------|------|------|------|
| **SISD** | Single Instruction Single Data | 单指令流单数据流 | 单处理器;可用流水线;串行标量计算机 | Intel 4004 |
| **SIMD** | Single Instruction Multiple Data | 单指令流多数据流 | 单(主)处理器;向量运算:一条向量指令 = 对数据流执行许多机器指令;可用流水化向量单元或处理器阵列实现 | CRAY-1、ILLIAC-IV、ICL DAP |
| **MISD** | Multiple Instructions Single Data | 多指令流单数据流 | 多处理器;单一数据流被多条指令流处理;流水线结构与脉动阵列(Systolic Array) | Colossus Mark II(1944) |
| **MIMD** | Multiple Instruction Multiple Data | 多指令流多数据流 | 多处理器;多条指令流作用于多数据流 | 多处理机(Multiprocessors)、多计算机(Multicomputers) |

**Flynn 分类的不足(P20)**:太宽泛(尤其对 MIMD);与 ILP 关系不清;未区分共享内存/分布式内存;与并行编程模型关系不明。

### 2.3 结构组成分类(Structural Composition)(P21–P26)

通用并行结构:处理器经互连网络(Interconnection Network)连接存储器。分类依据:**内存物理上位于何处、是否直连处理器、网络连通性**。

**P22–P23 列出的体系结构类别**:

- **向量机 / 处理器阵列(Vector Computer / Processor Array)**:串行处理器连接一串同步的处理单元 PE。
- **多处理机(Multiprocessors)**:共享内存的多 CPU 计算机。
  - **集中式多处理机(Centralized)**:一组处理器共享总线与同一物理内存 → **UMA(Uniform Memory Access)** → **SMP(Symmetric Multiprocessors)**。
  - **分布式多处理机(Distributed)**:内存分布在多个处理器上,但形成单一逻辑地址空间 → **NUMA(Non-uniform memory access multiprocessor)**。
- **多计算机(Multicomputers)**:每个处理器有各自独立的局部地址空间(disjointed)。
  - **非对称多计算机**:前端机(用户交互与 I/O)+ 后端机(并行任务)。
  - **对称多计算机**:所有计算机功能相同 → **集群(Clusters)与工作站网络(NOW)**。
- **大规模并行处理机 MPP(Massively Parallel Processing)**:上述各类的组合,计算部件数量达数百至数千,采用高速互连网络紧耦合。**所有[当前]超级计算机都属于此类**。

**并行体系结构类型汇总(P24–P26)**:

| 类型 | 特点 |
|------|------|
| Shared Memory Multiprocessor(SMP) | 共享内存地址空间;总线式或互连网络式存储系统 |
| Uniprocessor | 标量处理器 / 向量处理器 / SIMD 处理器 |
| Distributed Memory Multiprocessor | 节点间消息传递;MPP(大量处理器) |
| Cluster of SMPs | SMP 节点内共享内存,节点间消息传递;处理器数大时可视为 MPP |
| Multicore SMP + GPU Cluster | SMP 节点内共享内存,节点间消息传递,附 GPU 加速器 |
| Multicore | 多核处理器 / GPU 加速器 / "融合"型处理器-加速器(核可硬件多线程,即超线程) |

### 2.4 共享内存 vs 分布式内存 & UMA/NUMA(P62–P80,★重点)

**共享物理内存(P62)**:给单处理器系统加处理器,处理器共享系统资源(内存、存储等)。任一处理器可访问任一内存地址,任一 I/O 控制器可访问任一内存地址;单一物理内存地址空间;OS 可在任一/全部处理器上运行;通信通过 load/store 发生。

**SMP 的内存系统互连差异(P64–P69)**:

| 架构 | 内存组织 | 访问特性 |
|------|----------|----------|
| **总线式 SMP(Bus-based)** | 处理器共享总线访问内存模块 | **UMA**(内存非 cache 等距);可能有多个内存模块(物理地址空间交叉 interleaving);cache 引入数据一致性问题 → 需 cache 一致性硬件(**CC-UMA**) |
| **交叉开关 SMP(Crossbar)** | 为每个处理器/I/O 控制器复制内存路径 | UMA;带宽线性扩展(无共享链路);但增量成本高(多处理器难以承受);可用多级互连网络 |
| **"舞厅" SMP(Dance Hall)+ 共享 cache** | 互连网络连接处理器与集中式内存 | UMA;网络决定性能(从总线到交叉开关的连续谱);内存与处理器物理分离;共享 cache 减少一致性问题 |
| **NUMA SMP** | 内存物理上靠近各自处理器(分布式),但仍是共享 | 本地内存快,远端内存慢;性能依赖数据局部性(data locality);cache 一致性问题更严重;互连网络更可扩展 |

**Cache 一致性(P71–P79)**:

- **MESI 协议(P74)**:每个 cache line 有四态:
  - **M(Modified)**:本 cache 已修改,且不存在于其他 cache;需写回以保证内存一致性。
  - **E(Exclusive)**:自主存读出但尚未修改;其他 cache 无有效副本。
  - **S(Shared)**:已读未改;其他 cache 可能有有效副本。
  - **I(Invalid)**:本 cache line 无有效数据(常发生在 S 态被其他处理器请求独占所有权时)。
- **实现方式(P75)**:
  - **监听(Snoop)**:修改 CL 时,CPU 向全系统广播地址(总线是广播介质,cache 控制器监听所有事务)。
  - **目录(Directory)**:芯片组(网络)记录哪些 CL 在何处,过滤一致性流量;每个内存块有关联目录信息,记录各 cache 中副本状态。
- **False Sharing(P78)** ★:独立的多个数据元素恰好位于同一 cache line 上,每次更新都导致 cache line 在线程间来回搬运("slosh back and forth")。这是并行执行造成的 cache thrashing。

### 2.5 网络拓扑结构(P82–P89,★重点含计算)

**互连网络的四个特征(P85)**:
- **拓扑(Topology, what)**:网络图的互连结构。
- **路由算法(Routing Algorithm, which)**:限制消息可走的路径集合。
- **交换策略(Switching Strategy, how)**:数据如何遍历路由(电路交换 vs 包交换)。
- **流控机制(Flow Control, when)**:何时遍历,遇到流量如何处理。

**拓扑属性(P86)**:
- **路由距离(Routing distance)**:源到目的路径上的链路数。
- **直径(Diameter)**:最大路由距离。
- **平均距离(Average distance)**。
- **对分(Bisection)/ 对分带宽**:把网络分成相等两半时所需切除的最少链路数(相应带宽)。

**Mesh 类互连比较表(P88,★必考)**:

| 拓扑(16 节点) | 节点数 | 交换开关数 | 直径(Diameter) | 对分(Bisection) | 每节点边数(节点度) |
|----------------|--------|-----------|-----------------|------------------|---------------------|
| **2D Mesh** | 16 | 16 | **6** | 4 | 4 |
| **2D Torus**(环面) | 16 | 16 | **4** | 8 | 4 |
| **Iliac Mesh** | 16 | 16 | **3** | 8 | 4 |
| **4-Cube / Hypercube**(超立方体) | 16 | 16 | **4** | 8 | 4 |

**Tree 类互连比较表(P89,★必考)**:

| 拓扑 | 节点数 | 交换开关数 | 直径 | 对分 | 每节点边数 |
|------|--------|-----------|------|------|------------|
| **Binary Tree**(二叉树) | 8 | 15 | **6** | 1 | 3 |
| **4-ary Hypertree(depth 2)**(4 叉超树) | 16 | 28 | **4** | 8 | 4 |
| **Binary Fat Tree**(二叉胖树) | — | — | — | — | — |

> 说明:胖树(Fat Tree)通过越靠近根节点链路越"粗"(带宽越高)来克服普通树对分带宽低的缺陷,常用于现代超算互连(如 IBM Summit 使用 Dual-port Mellanox EDR InfiniBand FatTree,P108;SGI NUMAlink "fat tree",P101)。

### 2.6 三种并行编程模型(P109–P135,★重点)

编程模型由创建机器抽象视图的语言与库组成,需回答四个问题:**控制(并行如何创建、操作次序)、数据(私有/共享、共享数据如何访问)、同步(如何协调、原子操作)、成本**(P112)。

| 编程模型 | 控制/线程 | 数据 | 通信/同步 | 典型实现 |
|----------|-----------|------|-----------|----------|
| **共享内存(Shared Memory)** | 程序是线程集合,可动态创建;每线程有私有变量(栈)与共享变量(静态/堆) | 线程通过读写共享变量**隐式**通信 | 通过共享变量上的同步(锁、barrier)协调 | OpenMP、Pthreads |
| **消息传递(Message Passing)** | 程序是命名进程的集合,启动时固定;每进程有独立局部地址空间,**无共享数据** | 逻辑共享数据划分到各进程局部 | 通过显式 send/receive 通信,每次通信隐式含协调 | **MPI** |
| **全局地址空间(Global Address Space)** | 命名线程集合 | 局部 + 共享数据,但共享数据划分到各进程;远程数据昂贵 | 介于消息传递与共享内存之间 | UPC、Titanium、Co-Array Fortran;RDMA 硬件支撑 put/get |
| **数据并行(Data Parallel)** | 单一控制线程,由并行操作组成;并行操作施加于数据结构(数组)的全部/子集 | 通信隐含于并行算子;语句同步执行 | 协调隐式(同步执行);函数式(无副作用) | CUDA、OpenCL、ISPC;Matlab 的 `C = A + B` |

**抽象 vs 实现(P124,重要提醒)**:
- 可在硬件实现共享地址空间的机器上**实现消息传递抽象**("发消息"=拷贝到消息库缓冲区);
- 可在不支持共享地址空间的硬件上**用软件实现共享地址空间抽象**(将含共享变量的页标记为无效,缺页处理程序发网络请求)。
- **务必分清:什么是编程模型(用于编写程序的抽象),什么是硬件实现。**

**现代实践:混合编程模型(P134)**:
- 集群多核节点内用**共享地址空间**,节点间用**消息传递**(实践中极其常见)。
- 数据并行模型(CUDA/OpenCL)在核内扩展到多核,同时采用共享地址空间模型允许同核上线程通信;kernel 内支持共享内存式同步原语。

## 三、重点公式 / 定律

### 3.1 通信时间模型(P87)

$$
\text{Time}(n)_{s \to d} = \text{overhead} + \text{routing delay} + \text{channel occupancy} + \text{contention delay}
$$

其中 channel occupancy(信道占用时间):

$$
\text{occupancy} = \frac{n + n_h}{b}
$$

- $n$ = 消息字节数(message #bytes);
- $n_h$ = 头部字节数(header #bytes);
- $b$ = 通信链路位率(bitrate)。

### 3.2 网络拓扑度量(P86,★计算)

- **直径(Diameter)** = 网络中任意两节点间**最大**路由距离(链路跳数)。
- **对分(Bisection)** = 将网络节点集分成相等的两半、并使两半不连通,所需切除的**最少链路数**;对应**对分带宽**。

#### 典型计算(以 P88 数据为准,16 节点)

- **2D Mesh(4×4)**:边节点到对角节点的最远距离 = (4−1)+(4−1) = **6** → 直径 6;对分 4。
- **2D Torus**:每维成环后最远跳数减半 → 直径 4;对分 8。
- **Iliac Mesh**:每行末尾连到下行起始形成对角 wrap,直径 3;对分 8。
- **Hypercube(4-cube = $2^4$ 节点)**:直径 = 维数 = **4**(汉明距离最大值);节点度 = 维数 = 4;对分 = $2^{4-1}$ = 8。

#### 二叉树(8 节点,P89)

- 直径 = 从最左叶到最右叶 = 经根 → **6**(2×树高,树高 3)。
- 对分 = 切根的两条链路之一即可平分 → **1**(根的瓶颈)。

### 3.3 向量长度寄存器 VL(P40)

$$
\texttt{vadd.vv} \text{ with } VL = 10 \;\Longleftrightarrow\; \texttt{for}(i=0;\,i<10;\,i++)\; V1[i]=V2[i]+V3[i]
$$

- VL 可设到 MVL(如 32);向量长度 > MVL 时需分多条向量指令处理(strip-mining)。

## 四、例题 · 代码 · 执行过程(★重点)

### 例题 1:三种并行挖掘方式(P28–P60,★核心例题)

**题意**:给定顺序标量代码,考察如何利用并行。

```c
for (i=0; i < N; i++)
    C[i] = A[i] + B[i];
```

每次迭代展开为:`load, load, add, store`。

**方案 #1:ILP(指令级并行,SISD)(P29)**
- 可运行于:流水线处理器、乱序执行处理器(独立指令就绪即执行,不同迭代同时处于指令窗口,硬件动态展开循环 loop unrolling)、超标量或 VLIW 处理器(每周期取/执行多条指令)。

**方案 #2:数据并行(SIMD)(P30)**
- 识别:每次迭代相互独立。
- 思路:程序员/编译器生成 SIMD 指令,在不同数据上执行来自所有迭代的同一指令。
- 向量化代码:

```
VLD     A -> V1     ; 向量装载 A
VLD     B -> V2     ; 向量装载 B
VADD    V1 + V2 -> V3  ; 向量加
VST     V3 -> C     ; 向量存储到 C
```

- 最适合在 SIMD 处理器(向量机、阵列机)上执行。

**方案 #3:线程/任务级并行(MIMD/SPMD)(P59–P61)**
- 思路:为每次迭代生成一个线程,每线程做同样的事(数据不同)。
- 可在 MIMD 机器上执行;GPU 用 **SIMT** 模型执行。
- **Warp 执行过程(P59)**:Warp = 在同一 PC 上执行相同指令的一组线程(NVIDIA 把 32 个 CUDA 线程组成一个 warp;AMD 把 64/32 个线程组成一个 wavefront)。Warp 0 的执行轨迹为:`PC X`(load)→ `PC X+1`(load)→ `PC X+2`(add)→ `PC X+3`(store),整组线程同步推进 PC。

### 例题 2:SIMD vs SIMT 执行模型(P54,★辨析)

- **SIMD**:一条指令流(一串顺序的 SIMD 指令),每条指令对应多个数据输入(向量指令)。
- **SIMT**:多个指令流(标量指令)构成线程,这些线程**动态**构成 warp;一个 warp 处理多个数据元素。
- **SIMT 主要优点**:
  - 可独立处理线程(每线程可在任一标量流水线上单独执行 → MIMD 处理模式);
  - 可将执行相同指令流的线程组织成 warp → SIMD 处理模式,发挥 SIMD 优势。
- GPU 编程模型为 SPMD(用线程,每线程执行同一代码、操作不同数据,有独立上下文);warp 由硬件动态形成。

### 例题 3:多线程 SIMD 处理器结构(P56–P58)

- 每个 SIMD 处理器上有多个 warp;warp 有各自的指令指针(IP)。
- warp 阻塞时上下文切换极快(零开销切换);**必须有"足够多"的 warp 来避免 stall**(即每 block 需足够线程)。
- 关系:**WARPs => SMT(同时多线程),FUs => SIMD**(P57)。
- SM(Streaming Multiprocessor,P58):32 个 SP(标量处理器,即 "CUDA core"),SP 间共享 fast local 'shared memory'(16 或 64 KiB);全局内存在设备上。每个 SP 执行一个线程。

### 例题 4:共享内存求和 —— 数据竞争(P116–P117,★必考计算题)

**题意**:`static int s = 0;`,两线程分别累加 `s = s + f(A[i])`,其中 `A=[3,5]`,`f(x)=x^2`,`s` 初值 0。正确结果应为 $3^2+5^2=34$。

```c
// Thread 1
for i = 0, n/2-1
    s = s + f(A[i])
// Thread 2
for i = n/2, n-1
    s = s + f(A[i])

static int s = 0;
fork(sum, a[0:n/2-1]);
sum(a[n/2, n-1]);
```

**问题**:race condition(数据竞争)——两线程同时访问同一变量且至少一个是写,且未同步。

**执行过程展开(P117)** —— `s += f(...)` 不是原子操作,展开为:
```
compute f(A[i]) -> reg0       // 计算函数值放 reg0
reg1 = s                       // 读 s
reg1 = reg1 + reg0             // 加
s = reg1                       // 写回 s
```

两线程交错执行可能出现:
- 结果 34(运气好,顺序正确);
- 结果 9(Thread 1 算 9 写回后,Thread 2 读到 0 → 写 9;Thread 1 读到 0 之前…… 实际取决于交错);
- 结果 25。

> 讲义给出可能值:**34、9 或 25**。原子操作仅为读和写本身(看不到半个数),但 `+=` 不是原子。

### 例题 5:改进的求和代码(加锁)(P118,★代码)

利用加法结合律,把多数计算放在私有变量上,降低共享频率,但仍需对共享 `s` 的更新加锁。

```c
static int s = 0;
static lock lk;

// Thread 1
local_s1 = 0;
for i = 0, n/2-1
    local_s1 = local_s1 + f(A[i]);   // 私有累加,无竞争
lock(lk);                             // 进入临界区
s = s + local_s1;
unlock(lk);

// Thread 2
local_s2 = 0;
for i = n/2, n-1
    local_s2 = local_s2 + f(A[i]);   // 私有累加
lock(lk);
s = s + local_s2;
unlock(lk);
```

**问题:为何不在循环内加锁?** 答:在循环内加锁会极大增加同步开销(每次迭代都 lock/unlock),抵消并行收益;应先私有累加再单次加锁合并。

### 例题 6:消息传递求和(P119–P120,★代码)

`s = f(A[1]) + f(A[2])`,两处理器各自持有部分数据,通过 send/receive 通信。

**解 1(可能死锁)** —— 取决于 send/receive 是阻塞语义:
```c
// Processor 1
xlocal = f(A[1]);
send xlocal, proc2;
receive xremote, proc2;
s = xlocal + xremote;
// Processor 2
xlocal = f(A[2]);
receive xremote, proc1;     // 注意:Proc2 先 receive
send xlocal, proc1;
s = xlocal + xremote;
```

**解 2(对称)** —— 两处理器都先 send 后 receive:
```c
// Processor 1 / Processor 2 (相同模式)
xlocal = f(A[i]);
send xlocal, other;
receive xremote, other;
s = xlocal + xremote;
```

**思考(P120)**:send/receive 行为像电话系统(同步/ rendezvous)还是邮局(异步缓冲)?若多于 2 个处理器如何聚合(常用 reduce / tree sum)?

### 例题 7:数据并行 ISPC(P127,★代码)

```cpp
// main C++ code:
const int N = 1024;
float* x = new float[N];
float* y = new float[N];
// initialize N elements of x here
absolute_value(N, x, y);
```

```cpp
// ISPC code:
export void absolute_value(
  uniform int N,
  uniform float* x,
  uniform float* y)
{
  foreach (i = 0 ... N) {        // foreach 即 map:对每个元素并行施加循环体
    if (x[i] < 0)
       y[i] = -x[i];             // 函数体视为施加在每个元素上的函数
    else
       y[i] = x[i];
  }
}
```

要点:把循环体视作函数,`foreach` 是 map 操作。

### 例题 8:Gather / Scatter 通信原语(P129–P130,★代码)

**Gather(收集)**:按索引数组从输入收集元素。
```cpp
export void absolute_value(
   uniform float N,
   uniform float* input,
   uniform float* output,
   uniform int* indices)
{
   foreach (i = 0 ... n) {
      float tmp = input[indices[i]];   // gather:非常规访问
      if (tmp < 0)
         output[i] = -tmp;
      else
         output[i] = tmp;
   }
}
```

**Scatter(散射)**:按索引数组写回。
```cpp
export void absolute_value(
   uniform float N,
   uniform float* input,
   uniform float* output,
   uniform int* indices)
{
   foreach(i = 0 ... n)
   {
      if(input[i] < 0)
         output[indices[i]] = -input[i];   // scatter:按索引写
      else
         output[indices[i]] = input[i];
   }
}
```

**Gather 指令(P130)**:`gather(R1, R0, mem_base);` —— "根据 R0 指定的索引,从缓冲区 mem_base 收集到 R1"。
- AVX2(2013)支持 SIMD gather,但**不支持** SIMD scatter(需标量循环实现);
- AVX-512 才有 scatter 指令;
- GPU 硬件支持 gather/scatter,但相比连续向量的 load/store 仍是昂贵操作。

### 例题 9:简单求和问题分解(P113–P115)

**问题**:对数组 A 的每个元素施加函数 f,再求和。
- $s = \sum_{i=0}^{n-1} f(A[i])$
- 数据:A 在哪(单一内存?划分?)、各处理器做什么、如何协调得到单一结果?

**共享内存分解(P115)**:
- p << n 个处理器连接到单一内存;
- 给每个处理器分配 n/p 个数,各自计算"私有"结果与部分和;
- 收集 p 个部分和得到全局和。
- 两类数据:**逻辑共享**(原始 n 个数、全局和);**逻辑私有**(各函数求值;各部分和算哪类?)。

## 五、考点与易错点

1. **编程模型 ≠ 执行模型**:顺序程序可跑在乱序硬件上;SPMD 可跑在 SIMD 硬件(GPU)上。考试常考二者区别与反例(P14)。
2. **Flynn 四类全称**必须能写出英文全称和含义,典型机型(SISD→Intel 4004;SIMD→CRAY-1/ILLIAC-IV;MISD→Colossus Mark II;MIMD→多处理机/多计算机)(P16–P19)。
3. **UMA vs NUMA**:UMA 内存等距、访问时间相同(总线/交叉开关/舞厅式);NUMA 本地快远端慢、性能依赖数据局部性。NUMA 仍属共享内存,区别于分布式内存多计算机(各自独立地址空间)(P66–P69、P80)。
4. **MESI 四态**含义、监听 vs 目录两种实现;**False Sharing** 是高频考点:独立变量落在同一 cache line 导致乒乓效应(P74–P78)。
5. **拓扑度量计算**:直径 = 最大跳数;对分 = 平分网络最少切除链路数。注意 2D Mesh 直径 6、Torus 直径 4、Hypercube(4-cube)直径 4 对分 8、二叉树(8 节点)直径 6 对分 1。**二叉树对分带宽最低(根瓶颈),胖树用渐粗链路补救**(P88–P89)。
6. **SIMD vs SIMT**:SIMD 用向量指令编程;SIMT 用线程(SPMD),硬件动态组成 warp(32 线程)。GPU 编程用线程而非 SIMD 指令(P53–P54)。
7. **数据竞争(race condition)**:两线程访问同一变量且至少一写且未同步。`+=` 不是原子操作,展开为 load-add-store 三步,可得到 34/9/25 三种结果(P116–P117)。
8. **锁的正确用法**:私有累加 + 单次加锁合并,不要在循环内加锁(P118)。
9. **消息传递 send/receive 语义**:同步(电话式 rendezvous)还是异步(邮局式缓冲)决定是否死锁;多于 2 个处理器需 reduce(P120)。
10. **抽象 vs 实现**:消息传递可在共享内存硬件上实现(缓冲区拷贝);共享地址空间可在无共享硬件上用软件实现(页失效处理)。务必分清编程模型与硬件实现(P124)。
11. **Cache line 与局部性**:cache 行(V=64B/128B)整体传送;时间局部性、空间局部性;LRU/NRU/Random 替换;modified 行需写回(evict)(P10–P13)。
12. **Warp / wavefront 规模**:NVIDIA warp = 32 线程;AMD wavefront = 64/32 线程(P52)。需要足够 warp 避免 stall(P56)。
13. **Gather/Scatter 硬件支持**:AVX2 仅 gather;AVX-512 才有 scatter;GPU 支持但昂贵(P130)。
