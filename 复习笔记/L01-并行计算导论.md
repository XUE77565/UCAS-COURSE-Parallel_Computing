## 二、核心知识点

### 1. HPC 性能量纲(必背)

| 前缀 | 缩写 | Flop/s(每秒浮点运算) | Byte(字节) |
|------|------|----------------------|-----------|
| Mega | M | Mflop/s = 10⁶ flop/s | Mbyte = 2²⁰ ≈ 10⁶ |
| Giga | G | Gflop/s = 10⁹ flop/s | Gbyte = 2³⁰ ≈ 10⁹ |
| Tera | T | Tflop/s = 10¹² flop/s | Tbyte = 2⁴⁰ ≈ 10¹² |
| Peta | P | Pflop/s = 10¹⁵ flop/s | Pbyte = 2⁵⁰ ≈ 10¹⁵ |
| Exa | E | Eflop/s = 10¹⁸ flop/s | Ebyte = 2⁶⁰ ≈ 10¹⁸ |
| Zetta | Z | Zflop/s = 10²¹ flop/s | Zbyte = 2⁷⁰ ≈ 10²¹ |
| Yotta | Y | Yflop/s = 10²⁴ flop/s | Ybyte = 2⁸⁰ ≈ 10²⁴ |

- **Flop**:浮点运算,默认双精度。
- **Flop/s**:每秒浮点运算次数。
- **Bytes**:数据大小,双精度浮点数占 8 字节。
- 量级演进顺序:M → G → T → P → E → Z → Y,相邻相差 10³(或 2¹⁰)。

### 7. 并行 vs. 并发(Parallelism vs. Concurrency)

- **并发(Concurrency)**:"当两个或多个执行流能够同时运行时,即发生并发。" —— 是执行流的**属性**(property)。
- **并行(Parallelism)**:"在任务执行的某一时刻,该任务中独立子任务的最大数量。" —— 是执行流的一种**状态**(state)。
- **并行计算(Parallel Computing)** → 在并行计算机上对任务进行划分及其副作用(如存储一致性 memory consistency)。
- **并发计算(Concurrent Computing)** → 并发系统中任务之间的交互(如信号处理 Signal Handling)。

### 8. 并行性的类型(Type of Parallelism)

| 层次 | 名称 | 含义 |
|------|------|------|
| 应用层 | Job Level(作业级) | 作业之间/作业内重叠;Orchestrator:OS、程序员 |
| 应用层 | Task Level(任务级) | 不同代码段/不同循环迭代/函数调用;依赖数据依赖与程序划分;任务级 == 程序级;Orchestrator:编译器、程序员、OS |
| 硬件层 | Thread Level(线程级) | 线程:具有单一 PC(程序计数器)的指令序列,用于在不同处理器上运行程序的不同部分;Orchestrator:程序员或编译器 |
| 硬件层 | Instruction Level(指令级, ILP) | 指令之间(独立指令在多个硬件资源上运行,需多个 ALU/Adder/Multiplier/MAC 等);指令各阶段之间(指令流水线) |
| 硬件层 | Arithmetic & Bit Level(算术/位级) | ALU 单元内、浮点运算内部 |

**作业级并行(Job Level)示例(执行过程还原)**:
- Inter-Job(作业间):在不同 CPU 活动上重叠作业,例如 I/O 检索。
- Intra-Job(作业内):Time 1 → CPU 跑 Job1,同时 I/O 跑 Job2;Time 2 → CPU 跑 Job2,同时 I/O 跑 Job1。
- 要求:重复资源;在处理长 I/O 请求时切换作业;在等待 DMA 传输时执行独立计算。

### 9. 为什么需要并行(三大驱动)

- **功耗墙(Power Density Limits)**:时钟频率提高会同时拉高供电电压 V,产生立方效应,功率密度曾逼近核反应堆、火箭喷嘴、太阳表面。
  - 并发系统更省电:动态功耗 ∝ V²fC;提高 f 会拉高 V(立方效应);增加核数只线性增加电容 C。
  - 高性能串行处理器浪费功耗:投机执行、动态依赖检查、隐式并行性发现都在烧功耗。
  - 结论:晶体管可以更多,但串行处理器不能更快;单纯继续提高时钟频率(business as usual)行不通。
- **处理器革命(Revolution)**:芯片密度仍每 2 年翻一番,但**时钟频率不再提高**;取而代之的是**处理器核数翻倍**;功耗受控不再增长。
- **摩尔定律重新诠释**:每 2 年每芯片核数翻倍;时钟频率不再增长甚至下降;需要应对百万级并发线程;需要处理片间(inter-chip)与片内(intra-chip)并行性。

### 10. 并行编程为什么难(三大规律总结)

**规律 1**:
- 相同操作数的不同实现性能差异显著,**可达 100 倍**。
  - Cache 不命中开销 ≈ 计算操作开销的 100 倍。
  - 是否使用向量指令。
  - 是否用多核/众核。
- **操作数最小化 ≠ 性能最大化**(注意:讲义原文是"操作数的最小化 = 性能的最大化",意指减少冗余操作有利于性能)。
- 对遗产代码"免费获得加速比"的时代已经结束,必须开发更多并行性。

**规律 2**:
- 写快速运行的代码不易:细致考虑内存层次、熟悉向量指令、开发高效并行;需要**算法 + 编程 + 体系结构**三方专业知识。
- 快速代码通常较大,可能违反良好软件工程规范。
- 编译器经常爱莫能助:通常需要算法上的内在变化;自动并行/向量化仍是 NP 问题。
- 高性能通常**不可移植**。

### 11. 并行计算五原则(Principles of Parallel Computing)

1. **Finding enough parallelism**:寻找足够的并行性(Amdahl 定律)。
2. **Granularity**:粒度——每个并行任务应该多大。
3. **Locality**:局部性——移动数据代价大于算术运算。
4. **Load balance**:负载平衡——不要让上千个处理器等一个慢的。
5. **Coordination and synchronization**:协调与同步——安全共享数据。
6. **Performance modeling / debugging / tuning**:性能建模/调试/调优。

### 12. 现代机器中的"自动"并行性

- 位级并行(Bit level):浮点运算内部等。
- 指令级并行(ILP):每个时钟周期执行多条指令。
- 存储系统并行:访存操作与计算重叠。
- 操作系统并行:商用 SMP 上多个作业并行运行。
- 局限:对于极高性能需求,仍需用户自行识别、调度和协调并行任务。

### 13. 局部性、负载平衡与程序员分层

- **局部性(Locality)**:大存储慢、快存储小;存储层次平均上既大又快;并行处理器总体上拥有大而快的 cache;对"远程"数据的慢访问称为"通信";算法应主要在本地数据上工作。
- **内存术语**:
  - **Memory latency(访存延迟)**:处理器发出访存请求(load/store)到被存储系统响应所经历的时间,例如 100 cycles、100 ns。
  - **Memory bandwidth(访存带宽)**:存储系统向处理器提供数据的速率,例如 20 GB/s。
- **负载不平衡(Load Imbalance)**:部分处理器因并行性不足(某阶段)、任务大小不均而空闲;例子:适应"感兴趣区域"、树结构计算、非结构化问题。
  - **静态负载平衡**:开始前确定工作量并均分。
  - **动态负载平衡**:工作量动态变化时动态重平衡,如 work-stealing(工作窃取)。
- **两层程序员模型**:
  - **效率层(Efficiency Layer,20%)**:专家程序员编写实现 kernel 的库、框架、OS;追求峰值性能最大化。
  - **生产力层(Productivity Layer,80%)**:领域专家/非专家通过组合框架与库构建并行应用;尽可能隐藏机器/并行细节;为生产力牺牲一定性能。

## 三、重点公式 / 定律

### 1. 摩尔定律(Moore's Law)
- 原始(1965,Gordon Moore,Intel 联合创始人):半导体芯片的晶体管密度每约 **18 个月(1.5 年)翻一番**(`2X transistors/Chip Every 1.5 years`)。
- 重新诠释:每 **2 年** 每芯片核数翻倍;时钟频率不再增加(可能下降)。

### 2. 器件缩小的影响(Impact of Device Shrinkage)
设特征尺寸(晶体管尺寸)缩小为原来的 1/x(即缩小因子 x):
- 时钟频率提升 **x 倍**(因连线更短;实际小于 x,因功耗限制)。
- 单位面积晶体管数提升 **x² 倍**。
- 芯片面积(Die size)通常再增 ~x 倍。
- 芯片原始算力提升 **~x⁴ 倍**;其中 ~x³ 用于片内并行(ILP)与局部性(cache)。
- **结论:大多数程序无需改动,可获得 ~x³ 倍加速**。

### 3. 动态功耗公式
$$P_{\text{dyn}} \propto V^{2} f C$$
- V:供电电压;f:时钟频率;C:电容。
- 提高 f → 提高 V → **立方效应**(功耗急剧上升)。
- 增加核数 → 仅线性增加 C(更省电)。

### 4. Amdahl 定律(Amdahl's Law)
设 s 为应用中**必须串行执行的工作比例**,则 (1−s) 为可并行比例,P 为处理器数:
$$\text{Speedup}(P) = \frac{\text{Time}(1)}{\text{Time}(P)} \le \frac{1}{s + (1-s)/P} \le \frac{1}{s}$$
- 即便并行部分完美加速,整体性能仍受**串行部分**限制。
- 上限 Speedup ≤ 1/s:即使处理器数 P → ∞,加速比不超过 1/s。
- **例**:若 s = 1%(0.01),则最大加速比 ≤ 100;若 s = 10%(0.1),最大加速比 ≤ 10。

### 5. 性能鸿沟与扩展性(Scaling)
- **Processor-DRAM Gap**:µProc(微处理器)延迟每年改善约 60%,DRAM 每年改善约 7%;处理器-存储器性能差距每年增长约 50%。
- 内存密度每 3 年翻倍;处理器逻辑每 2 年翻倍。
- **强扩展性(Strong scaling)**:问题规模固定,增加处理器数。
- **弱扩展性(Weak scaling)**:问题规模随处理器数成比例增长。
- 开放问题:能否在并发翻倍的同时不让内存翻倍?

### 6. "定理"(性能差距定理)
给定数学函数 f 和实现并运行其上的处理器:
$$\frac{f\text{ 的最优实现性能}}{f\text{ 的简单直接性能}} \approx 10 \sim 100$$
- 同一函数的最优实现与朴素直接实现的性能差距可达 **10 到 100 倍**。

### 7. 并行开销(Overhead of Parallelism)
并行开销来源(每项在某些系统上可达毫秒级 = 数百万次 flop):
- 启动线程或进程的代价;
- 通信共享数据的代价;
- 同步代价;
- 额外(冗余)计算代价。
- **权衡(Tradeoff)**:算法需要足够大的工作单元(大粒度)以并行高效,但又不能大到没有足够并行工作。

## 四、例题 · 代码 · 执行过程(★重点)

### 例题 1:量纲换算与最快计算机性能解读

**题意**:讲义给出当前最快公开机器 El Capitan 性能为 1.742 EFlop/s。请据此换算并理解量级。

**已知**:
- E(Exa)= 10¹⁸。
- El Capitan:1.742 EFlop/s,11,039,616 个 CPU+GPU 组合核心。

**求解**:
- 1.742 EFlop/s = 1.742 × 10¹⁸ flop/s
- = 1742 PFlop/s
- = 1,742,000 TFlop/s
- = 1.742 × 10⁹ GFlop/s
- 对比:1964 年 CDC 6600 = 3 Mflop/s = 3 × 10⁶ flop/s。
- 性能比值:1.742 × 10¹⁸ / (3 × 10⁶) ≈ 5.8 × 10¹¹,即约 5800 亿倍提升。

**答案**:El Capitan 约每秒 1.742 × 10¹⁸ 次浮点运算,相当于 1742 PFlop/s,是 CDC 6600 的约 5800 亿倍。

### 例题 2:Amdahl 定律应用

**题意**:设某应用串行部分比例 s,使用 P 个处理器。求加速比上限,并讨论 s = 1% 时的极限。

**已知**:Amdahl 公式 Speedup(P) ≤ 1 / (s + (1−s)/P) ≤ 1/s。

**求解**:
- 上限:Speedup ≤ 1/s(无论 P 多大)。
- 当 s = 1% = 0.01:Speedup ≤ 1/0.01 = **100**(即使 P → ∞,至多 100 倍)。
- 当 s = 10% = 0.1:Speedup ≤ 10。
- 当 s = 0.5:Speedup ≤ 2。

**答案**:串行比例 1% 时,加速比上限为 100;10% 时上限为 10。Amdahl 定律说明"必须尽量减小串行部分",否则核数再多也徒劳。

### 例题 3:器件缩小因子的性能放大

**题意**:特征尺寸缩小因子 x 时,芯片原始算力提升多少?多数程序可获得多少加速?

**已知**:缩小因子 x。

**求解**:
- 时钟频率 ↑ x 倍;晶体管/单位面积 ↑ x² 倍;Die size ↑ ~x 倍。
- 原始算力 ↑ ~x⁴ 倍。
- 其中 ~x³ 用于片内并行(ILP)与局部性(cache),即"隐式并行"。
- **多数程序无需修改即可获得 ~x³ 倍加速**。

**答案**:原始算力 ~x⁴;典型程序 ~x³。

### 例题 4:DFT/MMM 性能优化案例(讲义案例 1、2)

**题意**:在 Intel Core i7(4 核,2.66 GHz,单精度)上比较朴素 C 代码与最优代码的 DFT 性能。

**已知**:
- 朴素"好" C 代码(约 1 KB) vs. 最优代码(约 1 MB)。
- 两种实现**操作数目几乎一样**;Intel 编译器、最优编译选项。
- 优化分解:
  - 内存层次结构优化:**5×**
  - 向量指令优化:**3×**
  - 多线程优化:**3×**

**求解**:
- 相同操作数下,小规模数据(16~512)约 **12×** 加速;大规模(如 1M)约 **35×** 加速。
- 矩阵乘法(MMM)案例:两种实现操作数完全一样(均为 2N³),性能差距同样显著。
- 由"定理":f 的最优性能 / f 的朴素性能 ≈ 10~100。

**执行步骤(DFT 性能优化分解)**:
1. 先做**内存层次结构优化** → 性能 ×5。
2. 再做**向量指令优化** → 性能 ×3。
3. 再做**多线程优化** → 性能 ×3。
4. 综合约 5 × 3 × 3 ≈ 45× 量级,与实测 12~35× 量级相符。

**答案**:操作数相同的两个实现性能可相差 10~100 倍;编译器很难自动完成这些优化,需手工优化(内存层次 + 向量 + 多线程)。

### 例题 5:获得高性能代码的抽象路径(图示还原)

讲义用"如何实现快速代码"的流程图说明难点。文字还原步骤:

1. **问题(problem)**:如 "Compute Fourier Transform"(计算傅里叶变换)。
2. → 经**算法理论(algorithm theory)** → 得到**最优算法(optimal algorithm)**:如 "Fast Fourier Transform (FFT)",复杂度 O(n log n) 或 4n log n + 3n。
3. → 经**软件开发(software developer)** → 得到**源代码(source code)**:一个 C/Fortran 函数。
4. → 经**编译器(compiler)** → 得到**快速可执行文件(fast executable)**。
5. 然后回到"这能工作得多好?",涉及**算法 / 软件 / 体系结构**三个层面的专业知识。
6. 结论:性能取决于算法、软件、编译、体系结构四个层次的协同。

### 例题 6:负载特征与体系结构适配(讲义"如何弥补鸿沟")

讲义用一张"计算访存比"示意图(不同算法 × 不同体系结构)。文字还原:

- **算法(按计算/访存比从高到低)**:深度学习 → 稠密线性代数 → 傅立叶变换 → 格子类算法 → 结构化网格(Stencil)→ 稀疏矩阵运算(SpMV)→ 图计算(Graph)→ 粒子类算法。
  - 高计算访存比:深度学习、稠密线性代数。
  - 低计算访存比:稀疏矩阵、图计算、粒子类算法。
- **体系结构**:Multicore、Accelerator、Heterogeneous(CPU/GPU/FPGA…)。
- **关键并行性维度**:Parallelism(vector, manycore…)、Memory(cache, coherence…)。
- **未来目标**:片上 1000+ 核、10 亿级并行度。
- **结论**:要弥补算法-机器鸿沟,需针对负载特征选择适配的体系结构。

## 五、考点与易错点

1. **量纲易错**:M/G/T/P/E/Z/Y 是 10³ 进位(或 2¹⁰);别把 Peta(10¹⁵)与 Exa(10¹⁸)混淆;Byte 是 2 的幂(Mbyte = 2²⁰),Flop/s 是 10 的幂(Mflop/s = 10⁶)——两套前缀混用,记忆时分开记。
2. **Linpack 三量**:Rmax(实测最大,用于排名)、Rpeak(理论峰值,上界)、Nmax(达 Rmax 时矩阵规模)。TOP500 按 **Rmax** 排序。
3. **第一"之最"清单(易混)**:
   - 第一台超级计算机:**CDC 6600**(1964);第一个 **RISC** 系统。
   - 第一台达到 **Tflops**:**ASCI Red**(1996,也是第一台用现成 CPU 制造)。
   - 第一台突破 **1-Petaflop**:**IBM Roadrunner**(2008,首台异构加速器机)。
   - 第一台突破 **1-Exaflop**(公开):**Frontier**(2022)。
   - 第一台"商用现成(commodity)"超算:**Cray-2**(1985);真正意义的 commodity 集群:**Beowulf**(1998)。
4. **Cray-1 关键参数**:136 **M**flop/s、80 MHz、8 个 64×64 向量寄存器、向量长度 VL + 向量掩码 VM —— 注意是 M(兆)不是 G。
5. **Amdahl 公式**:Speedup ≤ 1/(s+(1−s)/P) ≤ 1/s;**上限是 1/s 而不是 P**;串行比例 s 决定上限,而非核数。
6. **功耗立方效应**:功耗 ∝ V²fC,提 f 同时提 V → 立方增长;**增核只线性增 C**——这是转向多核的根本原因,不是"晶体管做不大"。
7. **器件缩小结论**:原始算力 ~x⁴,程序免费得 ~x³(靠 ILP + cache),不是 x⁴。
8. **"操作数最小化 = 性能最大化"**容易被反读;实质是减少冗余操作有利,但相同操作数下实现差异仍可达 100×。
9. **并行 vs. 并发**:并发是执行流的属性(可能同时),并行是执行流的状态(实际同时);两者不等价。
10. **强/弱扩展性**:强扩展——问题固定增核数;弱扩展——问题随核数等比增大。别混。
11. **自动并行/向量化是 NP 问题**——这是编译器"爱莫能助"的根因,常作简答考点。
12. **性能不可移植性**:快速代码通常依赖具体体系结构(向量指令、cache 层次),换平台即失效。
